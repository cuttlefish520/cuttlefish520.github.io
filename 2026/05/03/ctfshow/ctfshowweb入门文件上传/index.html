文件上传漏洞是指用户上传了一个可执行的脚本文件，并通过此脚本文件获得了执行服务器端命令的能力。“文件上传” 本身没有问题，有问题的是文件上传后，服务器怎么处理、解释文件。如果服务器的处理逻辑做的不够安全，则会导致严重的后果。

要点:用户上传可执行文件，服务器未对文件进行一个合理的检查过滤

MIME 验证的核心定义（大白话版）

MIME（Multipurpose Internet Mail Extensions）全称是「多用途互联网邮件扩展」，本质是文件的 “类型身份证” —— 用来告诉浏览器 / 服务器 “这个文件是什么类型”（比如是图片、文本、PHP 脚本）。
而「MIME 验证」就是：

服务器在接收文件（比如你上传的文件）时，检查请求头里的「MIME 类型」是否符合预期（比如只允许图片类型），不符合就拒绝接收。

举个直观例子：

你本地有个文件叫 1.jpg，但实际内容是 PHP 代码（<?php eval($_GET['cmd']); ?>）；
正常图片的 MIME 类型是 image/jpeg，PHP 文件的 MIME 类型是 application/x-httpd-php；
如果服务器开启 MIME 验证，只允许 image/jpeg 类型，你直接上传这个伪装的1.jpg，若请求头里的 MIME 类型是 application/x-httpd-php，就会被拦截；但如果把请求头的 MIME 类型改成 image/jpeg，就能绕过验证。

.user.ini文件（nginx）

.user.ini 是 PHP 针对每个目录的自定义配置文件—— 简单说就是：你可以在网站的任意目录下创建一个 .user.ini 文件，里面写 PHP 的配置指令，PHP 会自动加载这个文件的配置，覆盖 php.ini 的全局配置（仅对当前目录及子目录生效）。

.user.ini 里能写的配置指令有限，但以下两个是 的核心：

配置指令 作用 CTF 价值
auto_prepend_file = 文件名 执行当前目录下的 PHP 文件前，先自动包含（执行）指定文件 把非 PHP 文件（如.jpg）当作 PHP 执行
auto_append_file = 文件名 执行当前目录下的 PHP 文件后，自动包含（执行）指定文件 效果同上，只是执行时机不同

核心逻辑：auto_prepend_file 相当于给当前目录所有 PHP 文件加了一句 include "指定文件"，即使指定文件是 .jpg/.txt 后缀，也会被当作 PHP 代码执行。

新建一个.user.ini文件内容为auto_prepend_file=test7.png(我们上传的一句话木马图片)，然后再将其命名为.user.ini.png，先上传图片木马，再上传.user.ini.png文件同时抓包，再将请求包中文件名改回.user.ini发包。因为.user.ini只能对它所在的目录和子目录进行配置，所以需要先访问一下/upload让配置启动，然后再蚁剑链接，顺序不能错

getimagesize()

是 PHP 内置函数，官方定义是「获取图像尺寸信息」，但它的核心底层逻辑是： 读取文件的前几个字节（文件魔数 / 文件头），根据魔数判断文件类型，再尝试解析图像尺寸；如果不是图像格式，返回 false。 简单说：getimagesize() 验证的是「文件是不是图像」，但判断依据只有文件头的魔数，而非文件的实际内容、后缀名。读取文件前 6 个字节，判断是否匹配已知图像格式的魔数

CRC

核心作用是检测数据在传输 / 存储过程中是否被篡改或出错。

原理简单：基于二进制除法（模 2 运算），把数据当作一个大二进制数，除以一个预设的「生成多项式」，余数就是 CRC 值。
